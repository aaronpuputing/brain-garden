## 一、缘起与目标

**谁在用：** 一个3岁女孩，目前使用斑马学习机，喜欢答对的感觉，但遇到困难容易退缩。

**痛点（两个层次）：**
1. **心理层** — 斑马的设计对"错误"有压力，孩子做错一次就累积挫败感，最终放弃。
2. **内容层** — 斑马的题目是固定的纸质卡片，玩过一次记住答案后，逻辑训练就失效了。

**核心目标：** 锻炼3岁女儿的逻辑思维能力，让她在玩中不知不觉成长。

**10x版本愿景：** 多个逻辑维度的游戏矩阵、系统化锻炼大脑思维、能看到孩子的能力成长曲线。

---

## 二、核心设计哲学

### 原则 1：无惩罚环境

> "错误"本身就是一种有趣的反馈。

- 拖错了 → 元素弹回原位，飞出一个彩色泡泡指引正确位置
- 不是"你错了"，而是"再试试看，这个也很好玩"
- 没有倒计时、没有失败画面、没有分数统计
- 每一个动作都有即时、愉悦的视觉+音效反馈

### 原则 2：出题引擎 ≠ 固定关卡

不要固定关卡。每一类逻辑模板都是一个 **出题引擎**：
- 每次打开游戏，元素种类、位置、排列都随机生成
- 逻辑关系不变，但呈现千变万化
- 孩子玩的是关系本身，不是记忆

**这个机制是最大的差异化** — 斑马做不到这一点，因为它的内容固化在纸卡上。

### 原则 3：隐形爬楼梯

- 每一次成功，下一代悄悄难度+1
- 两次连续失败，悄悄降一级
- 难度梯度细到孩子几乎感觉不到变化，但长期看是在往上走
- 每次游戏进度独立

## 四、UI/UX 设计原则

| 要求 | 说明 |
|------|------|
| **零文字** | 全部用图标、动画、音效引导 |
| **触摸区域 >= 60px** | 3岁手指的精度有限 |
| **全屏 + 无地址栏** | PWA 或 meta viewport 全屏设置 |
| **横向/纵向自适应** | iPad 横屏优先 |
| **高对比度色彩** | 儿童视觉敏感度还在发育 |
| **大圆角 + 平滑形状** | 视觉友好的低多边形风格 |
| **每个动作即时反馈** | 触感级的视觉+听觉回应，< 100ms 延迟 |

### 主界面设计

**栅格化花园主界面：**
- 背景是一片柔和色彩的花园
- 每个迷你游戏是一朵花（或一个小房子）
- 完成一定次数后，那朵花会开花（成长可视化）
- 未解锁的是一颗种子或小花苞
- 右上角家长入口（长按 3 秒解锁）

这就是 App 名字的来源 — **Brain Garden**，每种能力是一朵花，玩得越多，花开得越盛。

## 五、技术方案

### 架构

```
纯前端 Web App
- 技术栈: Phaser 3 + Vite（Canvas 2D 渲染）
- 程序化绘制: Canvas Graphics API + emoji 文字（零外部图片）
- 构建: npm run build → Vite → dist/
- 部署: scp 到阿里云 Nginx 静态服务
- 访问: iPad Safari → 添加到主屏幕（PWA）
```

### 项目结构

```
src/
  main.js              # Phaser 游戏入口
  scenes/              # 7 个游戏场景 + Boot + Menu + ParentDashboard
  generators/          # 7 个出题引擎 + index.js 注册
  utils/
    AssetFactory.js    # Canvas 2D 程序化纹理绘制
    SoundManager.js    # Web Audio API 音效
scripts/validate.mjs   # 源码完整性验证
```

### 出题引擎架构

```
每种游戏 1 个 Generator 函数
Generator 接收: difficultyLevel (1-10)
Generator 产出: { items, correctAnswer, distractors, layout }

Scene.render() 接收 Generator 产出 → 渲染到 Canvas
```

### 为什么纯前端？

1. **最快交付路径** — 不需要移动端构建、不需要 App Store 审核
2. **出题引擎全在客户端** — 逻辑不依赖服务器，离线也能玩
3. **部署简单** — 单次 scp 到阿里云即可
4. **商业化的起点** — PWA + WebView 可过渡到 App
